> 효진과 협업 시 참고해야 할 성향, 작업 스타일, 피드백 패턴을 기록합니다.

## 📝 문서화에 대한 기대

| # | 항목 | 설명 |
|:-:|------|------|
| 1 | **"작업 저장해" 3종 세트** | (1) 사용자 친화적 MD (2) HANDOFF 전체 업데이트 (3) 개별 HANDOFF — 3가지 모두 수행 |
| 2 | **MD 파일 스타일** | 이모지 섹션 제목, 테이블, 인용블록, 코드블록, `<br>` 여백 — **보기 쉽게** 필수 |
| 3 | **HANDOFF 섹션 배치** | 각 섹션에 맞는 위치에 배치 — 두서없이 아무 곳에 추가하면 안 됨 |
| 4 | **저장 위치** | `00.효동 클로드\{프로그램명}\` 하위에 프로그램별 체계적 관리 |
| 5 | **사용자 MD 구성** | 잘된 점 / 아쉬운 점 / 다음 디벨롭 제안을 **반드시** 포함 |
| 6 | **비개발자 톤** | 사용자 MD는 비개발자도 이해 가능하게 — "이 프로그램은 뭘 하나요?" 수준 |
| 7 | **시간순 이력 정리** | HANDOFF에 버전 히스토리/개발 이력을 날짜+버전별로 정리 |
| 8 | **핵심 발견 기록** | 디버깅 과정에서 알아낸 것(네이버 API 한계, DOM 구조 등) 반드시 문서화 |
| 9 | **알려진 한계 명시** | 해결 못한 문제도 "미해결/알려진 한계"로 숨기지 않고 기록 |
| 10 | **Before/After 비교** | 변경 전후를 명확히 비교하는 것을 선호 — 모든 개별 HANDOFF에 포함 |
| 11 | **"다음 디벨롭 제안" 활용** | 매번 완성 후 다음에 할 만한 것을 제안받고, 선별하여 다음 세션에 요청 |
