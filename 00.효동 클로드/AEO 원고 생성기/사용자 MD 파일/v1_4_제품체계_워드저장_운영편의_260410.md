## ✏️ 시트가 이렇게 바뀝니다

```
이전:  A 발행일 | B 채널 | C 카테고리 | D 구조 | E 키워드 | F 담당자 | G 발행링크 | H 제목 | I 본문 | ...
현재:  A 작성일 | B 채널 | C 제품명 | D 카테고리 | E 구조 | F 키워드 | G 담당자 | H 발행링크 | I 제목 | J 본문 | ... | O 모델 | P 파일명 | Q 원고 다운로드
```

기존 데이터는 자동으로 한 칸씩 우측으로 옮겼습니다. 효진님이 수동으로 정리할 필요 없어요. 매핑 테이블(W~Y)도 그대로 보존됐습니다.

<br>

## 🔗 제품별 링크 자동 삽입 — 가장 큰 변화

이전엔 본문에 들어가는 자사몰 링크가 도메인 수준(`meditial.co.kr`)만 적혀서 효진님이 매번 정확한 URL로 수동 교체해야 했어요.

이제는:

- 시트 **T~V열의 매핑 표**에 효진님이 적어둔 자사몰·네이버 URL을 워커/웹이 매번 fresh로 읽음
- 원고 본문에 자연스럽게 링크가 들어가는 종류면 **정확한 URL 두 개** (자사몰 + 네이버) 모두 들어감
- 원고 종류상 링크가 어색하면 **아예 안 넣음** (Claude가 판단)

**ALL 원고는 더 똑똑하게**: 본문에 9개 제품이 다 등장하니까, 9개 매핑 표를 통째로 프롬프트에 주고 "본문에 등장하는 각 제품마다 그 제품의 개별 URL 사용"이라고 지시. 마지막 "브랜드 전체" 줄에는 메인 URL.

> 💡 자사몰 URL이 바뀌어도 시트만 수정하면 다음 원고부터 즉시 반영. 코드 수정 필요 없음.

<br>

## 📥 워드 저장 — 두 경로 모두

| **시트** | 워커가 처리 끝나면 Q열에 자동으로 생긴 `워드 다운로드` 클릭 → 즉시 다운로드 |

워드 파일 형식:

파일명: `AEO_blog_260410A11_all.docx` 같은 식. 채널이 워드프레스/카페 등이면 한국어 그대로(공백 제거): `AEO_워드프레스_260410B12_bc.docx`

## ⏳ 워커가 일하는 중인지 보이게

이전엔 시트에 입력하고 30~60초 동안 워커가 도는지 죽었는지 알 수가 없었어요. 이제는:

| J열에 보이는 것 | 의미 |
| --- | --- |
| (빈 칸) | 워커 차례 기다리는 중 |
| `⏳ 생성 중... (14:23:15 시작)` | **워커가 지금 만들고 있음** |
| (실제 본문) | ✅ 완료 |
| `❌ 자동 처리 실패 (3회 시도): ...` | ❌ 실패 (J열 비우면 재시도) |

⏳가 5분 이상 그대로면 워커가 죽었다는 신호 → J열 직접 비우면 다음 폴링에 다시 잡아요.

## 👍 잘된 점

1. **시트 한 곳에서 모든 게 해결** — 입력부터 진행 상황·완료·다운로드까지 시트만 봐도 됨. 클릭 수 최소화
2. **자사몰 URL 수정이 코드 배포 없이 즉시 반영** — 효진님이 시트 T~V열만 고치면 끝
3. **ALL 원고가 정말 똑똑해짐** — 9개 제품 등장하는 글에서 각각 정확한 URL이 들어감
4. **워드 저장 두 경로가 같은 빌더 공유** — 한 군데 고치면 양쪽 다 반영
5. **데이터 보호 완벽** — C열 삽입할 때 기존 데이터·드롭다운·매핑 테이블 다 자동으로 우측 이동, 손상 0건
6. **워커 진행 마커 + 픽업 보호 일석이조** — ⏳ 마커가 효진님 시각 확인용 + 다음 폴링 사이클 중복 픽업 방지를 동시에 해결
7. **채널별 글자수가 dict로 관리** — 나중에 새 채널 추가 시 한 줄만 추가하면 됨

6. **워커가 진짜 죽으면 ⏳가 영영 남음** — Railway 컨테이너 재시작 같은 상황에서 효진님이 수동 정리 필요
