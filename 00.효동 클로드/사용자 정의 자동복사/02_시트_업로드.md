## 3. 업로드 로직 (upload_data) — 핵심

### 전체 흐름

```
① A열 전체 읽기 (날짜 컬럼)
    ↓
② 첫 업로드 여부 확인
   └── A열 비어있으면 → 헤더 행 추가 (A1:R1)
    ↓
③ 같은 날짜 검색 (A열에서 date_str 매칭)
   ├── 있음 → 스마트 교체
   └── 없음 → 끝에 추가
```

### 스마트 교체 (같은 날짜 존재 시)

```
old_count = 기존 행 수 (같은 날짜)
new_count = 새 데이터 행 수
diff = new_count - old_count

if diff > 0:  (새 데이터가 더 많음)
    → old_end+1 위치에 diff개 빈 행 삽입
    → 뒷 데이터가 아래로 밀림 (보호)

if diff < 0:  (새 데이터가 더 적음)
    → old_start 위치에서 |diff|개 행 삭제
    → 뒷 데이터가 위로 당겨짐

→ A{old_start}:R{old_start + new_count - 1} 범위에 덮어쓰기
```

### 새 날짜 추가 (끝에 append)

```
next_row = len(a_col) + 1
→ 100행 배치로 업로드
→ A{next_row}:R{next_row + batch_size - 1}
```

## 6. 배치 업로드

```python
BATCH_SIZE = 100
for i in range(0, len(rows), BATCH_SIZE):
    batch = rows[i:i+BATCH_SIZE]
    ws.update(f"A{start}:R{start+len(batch)-1}", batch,
              value_input_option="USER_ENTERED")
    start += len(batch)
    print(f"  >> {i+len(batch)}/{len(rows)}행 업로드")
```

## 7. 평가

**잘된 점:**
- **행 삽입/삭제로 뒷 데이터 보호** — 핵심 요구사항 충족
- **A:R 범위 제한** — T+ 컬럼 안전
- **append_rows 미사용** — 피드백 반영
- **배치 업로드** — API 호출 효율화
- **헤더 자동 생성** — 첫 실행 시 편의

**부족한 점:**
- **업로드 후 검증 없음** — 실제 기록 데이터 대조 미구현
- **API 에러 시 부분 업로드 상태** — 롤백 없음
- **행 삽입/삭제가 순차적** — 대량 차이 시 느릴 수 있음
- **동시 접근 제어 없음** — 여러 사용자/프로세스 충돌 가능
