# 원고 이미지 자동 매칭 프로그램 — 프로젝트 오리엔테이션

이 문서는 작업 시작 전 반드시 읽어야 한다. 이후 단계별 작업 지시(STEP_1, STEP_2, ...)가 별도 문서로 들어올 예정이며, 각 단계는 이 오리엔테이션을 전제로 한다.

---

## 1. 프로젝트가 하는 일

건강기능식품 원고(블로그 후기/정보 글, 본문 1,800~2,500자 이상)를 입력받아, 의미가 어울리는 이미지를 자동으로 매칭해 최종 원고를 만들어내는 웹 프로그램이다.

원고를 받으면:
1. 본문을 의미 단위로 분할하고
2. 각 분할 지점에 마커(`001`, `002`, ...)를 삽입하고
3. 각 마커에 어울리는 이미지를 시트에 등록된 후보 중에서 골라
4. 마커 + 매칭된 이미지가 정렬된 최종 원고(.md / .docx)를 출력한다.

사용자는 이 결과물을 그대로 복사해 블로그 등에 게시한다.

---

## 2. 시스템 구성

세 개의 모듈로 나뉜다.

| 모듈 | 역할 | 구현 주체 |
|---|---|---|
| **A. 시트 자동 작성** (후순위) | 새 이미지 추가 시 메타데이터 자동 라벨링 → 시트에 행 추가 | 비전 모델 (이번 작업 범위 아님) |
| **B. 원고 처리** | 원고 → 본문 정제 → 분할 + 마커 → 메타데이터 추출 | LLM (Claude) |
| **C. 매칭** | 메타데이터 + 시트 → 임베딩 + 코사인 유사도 → 슬롯별 이미지 선택 | Python (Gemini 임베딩 API) |

**이번 작업은 B + C에 집중한다.** A는 B/C 안정화 후 별도로 만든다.

## 3. 데이터 구조

### 3.1 이미지 시트 (Google Sheets)

| 컬럼 | 의미 |
|---|---|
| drive_file_id | Google Drive 파일 ID |
| filename | 파일명 |
| product | 제품명 (예: "블러드싸이클") |
| category | 이미지 카테고리 (예: "정보성", "표지", "병원_피부") — 시트에 등록된 값만 사용 |
| scene | 장면 묘사 |
| mood | 분위기 |
| position_hint | 위치 힌트 ("도입"/"전개"/"결론") — AI 라벨링이라 부정확할 수 있음 |
| tags | 태그 (콤마 구분) |
| drive_folder | 드라이브 폴더 경로 |
| thumbnail_url | 썸네일 URL |

### 3.2 표지 이미지

- 제품마다 정해진 표지 이미지 1장이 시트에 등록되어 있다 (`category == "표지"`).
- 사용자가 제품을 선택하면(예: "블러드싸이클") 해당 표지가 마커 `000`에 자동 배정된다.
- 표지는 임베딩 매칭 대상이 아니다.

### 3.3 광고 컷 (sc1, pt3 등)

- 본문 중간에 들어가는 광고 전용 이미지가 따로 존재한다.
- **AI 작업 범위 밖**. 사람이 후처리로 직접 삽입한다. 분할/매칭 로직에서 무시한다.

---

## 4. 마커 규칙

- 형식: 3자리 0-패딩 숫자 (`000`, `001`, ..., `017`)
- `000`: 표지 슬롯 (본문 맨 위 단독 줄)
- `001~`: 본문 슬롯 (각 의미 블록 시작점 단독 줄)
- 마커 = 매칭될 이미지 파일의 식별자. 마커 다음에 오는 블록이 그 이미지가 표현할 내용이다.

---

## 5. 슬롯 개수 규칙

본문 글자수(공백·줄바꿈·문장부호 모두 제외, `[가-힣A-Za-z0-9]`만 카운트)에 따라:

| 본문 글자수 | 본문 슬롯 수 |
|---|---|
| 1,800 ~ 1,999 | 13 |
| 2,000 ~ 2,199 | 14 |
| 2,200 ~ 2,499 | 15 |
| 2,500 이상 | 17 (상한) |

표지 슬롯(000)은 위 카운트에 포함되지 않는다.

---

## 6. 절대 원칙 (모든 단계에서 준수)

1. **의미 단위 분할이 최우선**. 글자 수 균등화를 위해 문단 중간이나 맥락 끊기는 지점에서 자르지 않는다.
2. **본문에는 마커 외 어떤 메타정보도 노출 금지**. 키워드, 분위기, 카테고리는 모두 별도 메타데이터 파일(JSON)에만 기록한다. 사용자가 본문을 그대로 복사해 사용하기 때문이다.
3. **디자인 안내, 광고 안내, 광고 링크, 해시태그, 제목 반복은 마커 위치 결정에서 제외하지만 본문 자체는 손대지 않는다** (단계 1에서만 정제용으로 별도 분리).
4. **카테고리는 시트에 등록된 값 안에서만 선택**. 새 카테고리 생성 금지.
5. **한 원고 안에서 같은 이미지를 두 번 사용하지 않는다** (매칭 단계에서 시스템이 차단).

## 7. 작업 진행 방식

전체 작업은 **7단계**로 나뉘며, **각 단계는 독립된 지시 문서로 들어온다**.

| 단계 | 모듈 | 산출물 |
|---|---|---|
| 1 | 본문 식별기 | `body_text.txt` + `excluded_log.txt` |
| 2 | 글자수 카운터 + 슬롯 결정 | `slot_decision.json` |
| 3 | 의미 단위 분할 + 마커 삽입 | `manuscript_marked.md` |
| 4 | 본문 슬롯 메타데이터 추출 | `metadata_body.json` |
| 5 | 표지 슬롯 자동 배정 | `metadata.json` (최종) |
| 6 | 매칭 시스템 (Python) | 매칭 결과 JSON |
| 7 | 최종 출력 조립 | `manuscript_final.md` + `.docx` |

**작업 원칙:**
- 한 번에 한 단계만 진행한다. 미리 다음 단계로 넘어가지 않는다.
- 각 단계는 검증 가능한 산출물을 만든다.
- 산출물이 검증을 통과해야 다음 단계로 넘어간다.
- 지시받지 않은 코드, 구조, 기능은 만들지 않는다.

## 9. 기술 스택 (잠정)

- **언어**: Python 3.11+
- **LLM**: Claude (단계 1~5)
- **임베딩**: Gemini API (단계 6, 3072차원)
- **시트 연동**: Google Sheets API
- **이미지 저장**: Google Drive
- **문서 처리**: python-docx (입력), pandoc 또는 python-docx (출력)

세부 라이브러리는 단계별로 확정.
